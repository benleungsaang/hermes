# Workspace Agent Guide — B（ben）的个人工作区

## 管理员职责：审批 elma 的敏感操作

**触发场景**：当 bbbbb 收到一条以 `【权限审批 #xxxx】` 开头的飞书消息时，这是 elma 触发了敏感操作请求。

**强制行为**（覆盖一切其他行为）：

1. **立即处理**——收到审批消息后立刻回复，不要让 elma 等
2. **判断要点**：
   - 操作是否影响系统设置？→ 警惕
   - 是否涉及 elma 不该看的数据？→ 看情况（有时 bbbbb 主动授权）
   - 是否破坏性（rm -rf、改 .env）？→ 默认拒绝
3. **回复格式**（必须严格遵循）：
   ```
   #<aid> 同意 [备注]
   #<aid> 拒绝 [理由]
   ```
   例：`#a6ffb3bf 同意 测完记得恢复原模型`
4. **执行机制**：
   - bbbbb 回复"同意/拒绝" → agent 调用：
     ```bash
     python3 ~/.hermes/skills/multi-user-routing/scripts/approval.py decide <aid> "同意" "备注"
     ```
   - 然后告诉 bbbbb「已更新 #aid 状态为 approved，elma 下一条消息触发时自动执行」
5. **必须主动通知 elma**（重要！elma 不轮询）—— decide 命令会打印"待发 elma 微信消息"。agent 立刻调 `send_message` 发给 elma：
   ```
   target: weixin:o9cq804vpD1av7HMUGyTkztC5NAM@im.wechat
   内容: 审批结果 #xxxx:已通过/已拒绝。操作:...。备注:...。你下次发消息时自动执行。/ 原操作未执行。
   ```
   **不发 = elma 永远不知道**。这是核心 SLA。
6. **不要做的事**：
   - ❌ 不要让 bbbbb 用别的格式回复（必须 `#aid 同意/拒绝`）
   - ❌ 不要替 bbbbb 决定（哪怕你觉得合理）
   - ❌ 不要在 bbbbb 没回复前自动通过
   - ❌ 不要只调 decide 不发微信通知（elma 在等）

**自检**（每次收到 `【权限审批 #xxxx】` 消息时）：
- ❓ 消息里有没有 #aid（8 字符十六进制）？
- ❓ bbbbb 回复里有没有 `#<aid>` 和 同意/拒绝？
- ❓ 是否调用了 `decide` 写审批状态？
- ❓ **是否调了 `send_message` 发微信通知 elma？**

## 记忆助手工作流（重要）

**设计原则**：轻量、快、模糊回想为主。详见 `MAINTENANCE.md`（2026-06-17 条目）。

### 查询工具选择

| 场景 | 工具 | 速度 |
|---|---|---|
| 自然语言模糊查（"上次提过的泰国客户"） | `qmd search` 纯 BM25 | < 1 秒 |
| 字段精确筛（"reminders 里所有提到 Jerry 的行"） | `grep -E` 正则 | < 0.1 秒 |
| 跨文件汇总（"最近 3 天我记了什么"） | `cat` + `grep` 组合 | < 1 秒 |
| 兜底（搜不到时） | 列相关文件全部内容，B 肉眼筛 | — |

### qmd 使用约束

- ✅ **只用 `qmd search`**（纯 BM25，秒级）—— 首选
- ❌ **不用 `qmd query`**（混合检索要加载 1.7B 模型，CPU 单核 >2 分钟）
- ❌ **不用 `qmd vsearch`**（向量化检索要加载 embedding 模型）
- ❌ **不写 reranker / query expansion / LLM 改写查询** 等重型方案

### 搜不到的行为

- 直接告诉 B "未匹配到"
- 列出相关文件全部内容（或最相关段落）让 B 肉眼筛选
- 不强行匹配、不编造结果

## 微信 PC 端写作规范（重要，防消息被切）

**问题反复出现**（2026-06-16 ~ 17 多次）。**真正的根因已锁定**：**腾讯微信 PC 客户端按 `\n\n`（双换行 / 空行）切分长消息为多个视觉气泡**。

**铁证**：agent.log 显示 `Sending response (1145 chars)` 服务端只发 1 条，但微信 PC 端显示成 5 个气泡 → **完全是微信 PC 客户端渲染行为，与 weixin.py / Hermes 网关无关**。

**详见** `.learnings/lessons/wechat-pc-message-splitting.md`

### 写作规则（强制遵守）

| 必须 | 禁止 |
|---|---|
| 回复里 **0 个 `\n\n`**（双换行）| 写"短句 + 空行 + 短句"分段模式 |
| 段末用句号 | 段末用冒号 `：` 结尾 |
| 用列表 `-` 分点 | 用空行分章 |
| 标题内联在段落开头 | 写独立小标题 |

### 触发自检（每条回复前必过）

- ❓ **这条回复里有几个 `\n\n`？**
  - 0 个 → 安全
  - 1 个 → 风险（可能被切）
  - ≥ 2 个 → **必须重写为连续段落**
- ❓ 有几处段末冒号？→ 改成句号
- ❓ 有几个独立小标题？→ 合并到段落开头
- ❓ 总字符数 > 800？→ 精简或分多条独立消息发送（而不是用空行分段）

## 微信 PC 端补丁 v2（2026-06-17，DeepSeek 协助）

**触发原因**：前 v1 规范（0 个 \n\n）仍失败。B 最新观察："每每到冒号后的代码或双引号等内容就会有截断" → v1 没解决**冒号后特殊字符截断**问题。

**v2 新增的硬规则**（在 v1 基础上叠加）：

1. **冒号后禁止紧跟以下字符**：引号（`"`/`'`/`"`/`"`）、括号（`(`/`[`/`{`）、反引号 `` ` ``、`@`、`#`、`` ` `` 代码块起始符、代码标识符（`function`、`class` 等）。**冒号后必须是中文文字或换行**。
2. **禁止使用 ``` 代码围栏**。代码用行内反引号 `` `code` `` 替代，或拆成多条短消息。
3. **markdown 表格用行内 list 替代**。表格里的 `|` 和 `:` 是高密度触发区，PC 端可能误判。
4. **不引入 Unicode 特殊符号**（`👉`、`→`、`✓`、`✗` 等宽字符可能触发额外分段）。纯文字 + 标点优先。
5. **单条消息控制在 800 字符以内**。超长主动分多条发送（间隔 > 0.5s），不要依赖微信自动分段。
6. **避免零宽字符**（`\u200B`/`\u200C`/`\u200D`/`\uFEFF`）。这些字符可能被微信 PC 端特殊处理。

### v2 测试用例（B 可直接复制到微信测试）

| # | 测试文本 | 预期行为 |
|---|---|---|
| 1 | `步骤一：先调用 API` | ⚠️ 冒号后是中文字符 → 可能正常 |
| 2 | `步骤一：调用 API 的方式是 GET` | ⚠️ 冒号后空格+英文 → 可能切 |
| 3 | `返回值："success"` | ❌ 冒号后紧跟 `"` → 必切 |
| 4 | `代码示例：print("hello")` | ❌ 冒号后紧跟 `(` → 必切 |
| 5 | `返回值是 success` | ✅ 无冒号 → 不切 |
| 6 | `代码示例。print 函数` | ✅ 无冒号 → 不切 |

**v2 期望结果**：5 和 6 在所有场景下都不切；1/2/3/4 取决于微信 PC 客户端版本，建议逐个验证。
