## [2026-06-17] 记忆助手设计结论（B 已确认）

**需求定位**：B 想要的是"**快速模糊回想**"（"上次提过的泰国客户那种"），而非严格结构化字段查询。

**轻量化设计原则**（核心）：
- ✅ **速度优先**：查询 < 1 秒；建库/索引 < 1 分钟
- ✅ **简单工具链**：`grep` / `qmd search`（纯 BM25） / 简单 SQL 即可，**不写复杂 Python 工具**
- ❌ **不建新数据库**：继续用现有 .md 文件（reminders.md / corrections.md / domains/*.md 等）
- ❌ **不引入重型方案**：不写 reranker、不做 query expansion、不做 LLM 改写查询
- ✅ **搜不到就直接说**：明确告诉 B "未匹配到"，并列出相关文件全部内容让 B 肉眼筛选

**查询策略（按场景）**：
| 场景 | 工具 | 预期速度 |
|---|---|---|
| 自然语言模糊查（"上次提过的泰国客户"） | `qmd search` BM25 | < 1 秒 |
| 字段精确筛（"reminders 里所有提到 Jerry 的行"） | `grep -E` 正则 | < 0.1 秒 |
| 跨文件汇总（"最近 3 天我记了什么"） | `cat` + `grep` 组合 | < 1 秒 |
| 兜底（搜不到时） | 列相关文件全部内容 | B 肉眼筛 |

**不实施的功能**：
- ❌ cron 自动推送（"今天 W-XXX 到期"主动发微信）—— B 没要求
- ❌ SQLite 结构化数据库 —— 不必要
- ❌ 自动编号系统暴露给 B —— 编号是 Hermes 内部用的，B 用人话查

**实施位置**：
- 详情已写入 `AGENTS.md` 的"记忆助手工作流"章节
- 不再在 reminders.md 跟踪此事项（设计已完结，进入维护态）
